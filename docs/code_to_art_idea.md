# 프로젝트 기획: Code2Art (서비스 타이틀: Is My Code Art?)

## 1. 프로젝트 개요
- **목표**: 사용자가 작성한 단일 소스 파일(코드)을 분석하여, 코드의 논리적 아름다움을 **미술관에 걸릴 법한 모던 아트풍의 한 장의 그림 작품**으로 렌더링하는 서비스.
- **핵심 가치**: 정보의 단순한 차트나 그래프화(Data Visualization)가 아닙니다. 코드를 물감과 붓터치로 치환하여 개발자들에게 **"자신의 지적 결과물이 아름다운 조형적 예술로 승화되는 경험"**을 제공합니다. 이는 액자로 만들어 걸어두고 싶거나 SNS에 자랑할 수 있는 감성적 자극을 극대화합니다.

## 2. '코드의 아름다움' 평가 기준 및 시각화 매핑 (시각화 로직)
단순한 코드 포맷팅을 넘어, 소프트웨어 공학적인 관점에서 코드의 품질을 평가하고 이를 미술적 요소로 치환합니다.

### A. 제어 흐름 복잡도 (Cyclomatic Complexity)
- **개념**: `if`, `for`, `while`, `switch` 등의 분기문과 반복문이 얼마나 깊게 중첩되어 있는지 평가합니다.
- **시각화 (모던 아트)**: 선의 교차, 각도, 날카로움.
  - **Good (낮은 복잡도)**: 부드러운 곡선, 수평/수직이 잘 정돈된 선 (몬드리안 스타일의 안정성).
  - **Bad (높은 복잡도)**: 거칠고 날카로운 지그재그 선, 복잡하게 뒤엉킨 뾰족한 선형들.

### B. 모듈화 및 구조적 여백 (응집도)
- **개념**: 함수가 단일 책임 원칙을 잘 지켜 적절한 크기로 쪼개져 있는지 평가합니다.
- **미술적 렌더링 (여백의 마술)**: 캔버스 내에서의 밀도와 **여백의 미(White Space)**.
  - **Good**: 캔버스 위에 적절한 빈 공간이 함께 호흡하여, 마치 절제된 젠(Zen) 아트나 미니멀리즘 페인팅처럼 숨통이 트이는 편안한 구도.
  - **Bad**: 캔버스 전체를 검게 칠하거나 빈틈없이 물감 덩어리로 꽉 채워, 묵직하고 숨 막히는 압도적인 덩어리(God Object)를 형성.

### C. 일관성과 코드 스멜 (Consistency & Code Smells)
- **개념**: 네이밍 컨벤션의 일관성, 매직 넘버 사용 여부, 불필요한 코드 중복 존재 여부를 파악합니다.
- **시각화 (모던 아트)**: 색상 팔레트의 조화로움과 노이즈(얼룩).
  - **Good (높은 일관성)**: 명도와 채도가 잘 계산된 조화로운 컬러 팔레트 사용 (바우하우스 스타일의 세련된 색채).
  - **Bad (일관성 부족, 코드 스멜)**: 눈을 피로하게 하는 불균형적인 보색 대비의 남발, 배경 곳곳에 흩뿌려진 시각적 노이즈나 얼룩(불순물).

### D. 의존성 및 결합도 (Coupling)
- **개념**: 모듈(함수, 클래스) 간의 참조나 전역 변수 의존이 얼마나 복잡하게 얽혀 있는지 파악합니다.
- **시각화 (모던 아트)**: 도형들을 물리적으로 잇는 선(Edge) 또는 물감의 섞임.
  - **Good (낮은 결합도)**: 도형들이 독립적인 오브제로 깔끔하게 띄워져 있거나 얇고 투명한 실선으로만 연결됨.
  - **Bad (높은 결합도)**: 서로 다른 도형(색상)의 경계가 무너지며 탁하고 칙칙하게 섞이는 그라데이션 효과, 모든 도형을 관통하는 굵고 거무튀튀한 거미줄.

## 3. 사용자 커스터마이징 (화풍 선택)
사용자는 코드를 예술로 변환할 때, 최종적인 "모던 아트 스타일"을 선택할 수 있습니다.
- **기하학적 추상 (Geometric Abstract)**: 직선과 다각형 위주의 구조적이고 딱 떨어지는 형태 (예: 몬드리안, 말레비치).
- **서정적 추상 (Lyrical Abstract)**: 부드러운 유기적 곡선과 원형, 수채화처럼 번지는 느낌 (예: 칸딘스키).
- **액션 페인팅 (Action Painting)**: 물감을 흩뿌린 듯한 거칠고 역동적인 붓터치와 선의 궤적 (예: 잭슨 폴록).

## 4. 로직 구현 시 '예술성'을 극대화하는 접근 방향
데이터의 딱딱한 도식화 느낌(예: UML 다이어그램, 노드 트리 렌더링)을 완전히 배제해야 합니다.
1. **코드 추상 구문 트리(AST) 분석**: 코드를 분석해 수치화된 배열(DNA)을 뽑아냅니다.
2. **제너러티브 아트 엔진 (p5.js + WebGL)**: 
   - 노드나 선분 등에 Randomness와 Perlin Noise(자연스러운 굴곡을 만드는 노이즈)를 섞어, '컴퓨터가 그린 딱딱한 선'이 아니라 '사람이 붓으로 그은 듯한' 유기적 입자감을 생성합니다.
   - 캔버스 텍스처(종이, 캔버스 질감) 필터를 입혀 진짜 그림 같은 퀄리티를 냅니다.
3. **AI (Stable Diffusion) 렌더링 결합 (Phase 2)**: 가장 압도적인 예술성을 위한 최종 형태. 브라우저에서 1차로 생성된 기초 뼈대(스케치나 도형)와 추출된 메타데이터를 AI 엔진에 전달(Image-to-Image 방식)하여 실제 오일 페인팅이나 수채화 같은 거장의 터치로 최종 렌더링합니다. 이 방식은 보안 문제를 완벽히 해결(원본 코드가 아닌 스케치 이미지만 서버 전송)하면서도 예술성은 극대화할 수 있습니다.

### 💡 현재 진행 단계 (PoC - Proof of Concept)
초기 PoC 버전에서는 **"AI 렌더링 파트는 일단 제외"**하고, **100% 브라우저 내에서 구동되는 자체 제너러티브 아트 엔진(p5.js 등)**만을 활용하여 예술적 시각화의 가능성을 증명하는 것으로 개발을 시작합니다.

### B. 다중 파일 / 프로젝트 단위 분석 (추후 확장 타겟)
- **개념**: 사용자의 GitHub 저장소(Repository) 링크를 입력받아 프로젝트 전체의 종속성 트리와 모듈 간 상호작용을 분석합니다.
- **장점**:
  - **압도적이고 거대한 예술 작품**: 수백 개의 파일이 의존성 네트워크를 이루는 모습은 그 자체로 우주나 거대한 지도 같은 경이로운 스케일의 추상화(예: 노드-링크 다이어그램의 예술적 진화)를 만들어 낼 수 있습니다.
  - **진정한 의미의 '결합도' 평가**: 패키지 간 순환 참조 구조나, 공통 유틸리티의 방사형 의도 등을 완벽하게 잡아낼 수 있습니다.
- **단점**: 분석 시간이 오래 걸리고(AST 추출 병목), 사용자가 깃허브 권한을 연동해야 하는 허들이 생겨 초기 유입 스펙으로는 무겁습니다.

## 6. 보안 및 프라이버시 (코드 유출 방지 전략)
개발자들은 회사나 개인의 코드를 외부에 노출하는 것에 매우 민감합니다. 이 우려를 완전히 해소해야 서비스가 널리 쓰일 수 있습니다.

### A. 100% 브라우저(클라이언트) 기반 분석 (권장)
- **개념**: **백엔드 서버로 코드를 전송하지 않습니다.** 사용자의 브라우저 내에서 웹어셈블리(WebAssembly)나 자바스크립트용 AST 파서(예: Acorn, Babel, tree-sitter-wasm 등)를 구동하여 코드를 분석합니다.
- **장점**: 
  - 코드가 사용자의 기기를 절대 벗어나지 않으므로 **보안 우려가 0 (Zero-knowledge)**에 수렴합니다.
  - 서버 비용을 획기적으로 절약할 수 있습니다.
- **아키텍처**: 브라우저에서 코드를 분석 -> 추출된 메타데이터(수치)만 이용해 p5.js/WebGL로 클라이언트에서 직접 아트 렌더링.

### B. 메타데이터(파라미터)만 전송하는 하이브리드 방식
- **개념**: 만약 고도로 복잡한 AI(Stable Diffusion) 렌더링을 위해 서버의 그래픽 스펙이 반드시 필요한 경우 사용합니다.
- **방식**: 
  - 브라우저에서 먼저 코드를 파싱합니다.
  - 원본 코드 대신, 원본 코드를 파괴하여 얻은 순수 수학적 파라미터(예: `[복잡도: 0.8, 함수개수: 5, 응집도: 0.3...]`) 묶음만 안전한 HTTPS 통신으로 서버에 넘깁니다.
  - 서버는 이 숫자의 나열만 보고 예술 작품을 렌더링해서 돌려줍니다. 
  - 이 과정에서 서버는 원본 코드를 역추적할 수 없습니다.

### C. 강력한 보안 보증 마케팅
- **선언문**: 랜딩 페이지 메인에 "We do NOT store or send your code. 100% Local processing" 이라는 배지를 가장 크고 신뢰감 있게 보여주어야 합니다.
- **오픈소스 공개**: 클라이언트 측의 코드 파싱(AST 변환) 로직을 아예 오픈소스로 공개하여, "우리가 진짜로 서버에 안 보낸다"는 것을 투명하게 증명합니다.

## 7. 초기 PoC 기술 스택 제안 (React 기반 웹)
보안 우려를 없애고 빠른 프로토타이핑을 위해 **100% 프론트엔드(Client-side) 기반 아키텍처**로 구성합니다.

### A. Core Framework (웹 애플리케이션)
- **React (Next.js 권장)**: 사용자 UI 구성, 코드 입력 폼, 렌더링 결과 상태 관리 등을 위해 사용합니다. SEO와 초기 로딩 속도, 향후 서버리스 API(Next API Routes) 확장성을 고려할 때 순수 React(Vite)보다 Next.js를 추천합니다.
- **TypeScript**: 코드 품질 분석 도구 특성상, 안정적인 타입 정합성과 유지보수를 위해 필수적입니다.
- **Tailwind CSS + Framer Motion**: 모던하고 아티스틱한 랜딩 페이지 UI와 부드러운 애니메이션(예: 코드를 분석하는 동안 흐르는 파티클 효과) 구현을 위해 사용합니다.

### B. Code Parsing & Analysis (핵심 분석 로직)
- **Babel (Standalone) 또는 Acorn**: 사용자가 입력한 JavaScript/TypeScript 코드를 브라우저 위에서 직접 AST(Abstract Syntax Tree)로 파싱합니다. 웹팩이나 서버 없이 브라우저 메모리 안에서만 동작합니다.
- **escomplex (또는 자체 메트릭 추출기)**: 파싱된 AST 노드를 순회하며 Cyclomatic Complexity(순환 복잡도), Halstead 메트릭, 라인 수 등을 클라이언트에서 직접 계산합니다.

### C. Generative Art Visualization (렌더링 엔진)
- **p5.js (react-p5)**: 프로세싱(Processing) 로직을 웹에서 쉽게 렌더링할 수 있는 가장 대중적이고 강력한 제너러티브 아트 라이브러리입니다. 코드 메트릭스 시드(Seed)값에 반응하여 도형, 곡선, 파편들을 그립니다.
- **WebGL / Three.js (대안)**: 2D 캔버스를 넘어 3D 입체 공간에 코드를 둥둥 띄우거나 공간감이 들어간 파티클 아트를 구상한다면 가장 좋은 선택지입니다.

### 💡 PoC 렌더링 파이프라인 흐름
1. `Next.js` 코드 에디터 UI에 유저가 코드를 붙여넣음.
2. `Babel/Acorn`이 브라우저에서 즉시 코드를 파싱하여 AST 생성.
3. `AST`를 순회하여 복잡도, 모듈화 점수 등 [DNA 수치 배열] 추출.
4. 추출된 수치를 `p5.js` 캔버스 컴포넌트의 파라미터로 전달하여 Seed로 활용.
5. `p5.js`가 해당 규칙에 맞춰 시각적 입자/도형을 동적 렌더링 및 애니메이션화.
6. 결과물을 `이미지(PNG/JPEG)`로 내보내기 및 SNS 공유 로직 구현.

## 8. UI/UX 디자인 컨셉 (귀여운 로봇 테마)
사용자가 자신의 소중한 코드를 맡길 때 심리적 편안함을 주면서도 위트 있는 경험을 제공하기 위해 **"귀여운 아트 로봇"** 컨셉을 메인 테마로 잡습니다.

### A. 메인 캐릭터 (가칭: 코디-Cody)
- **비주얼**: 베레모를 쓰거나 팔레트/붓을 들고 있는 작고 동글동글한 로봇 캐릭터. 
- **역할**: 사용자가 코드를 입력하면 이 로봇이 돋보기를 들고 코드를 이리저리 관찰하는 애니메이션이 나옵니다.
  
### B. 인터랙션 및 애니메이션 요소
- **분석 중 (Loading)**: 
  - 코디가 땀을 뻘뻘 흘리며 코드를 읽거나, 머리 위의 전구가 반짝이며 고민하는 귀여운 로딩 애니메이션.
  - "코드의 냄새(?)를 맡아보는 중입니다...", "복잡한 IF문 사이를 탐험 중..." 같은 위트 있는 상태 메시지 노출.
- **그림 그리기 (Rendering)**:
  - 분석이 끝나면 텅 빈 캔버스 앞에서 코디가 붓글씨를 쓰거나 마구 물감을 뿌리는 액션(가상)을 취하고, 그에 맞춰 `p5.js` 캔버스 위로 선과 색이 촤르르 펼쳐집니다.
- **결과 발표**: 
  - 완성된 그림에 따라 로봇의 표정이 바뀝니다. 코드가 아름답다면 감명받아 눈물을 흘리거나 박수를 치고, 스파게티 코드라면 안경이 삐뚤어지며 어지러워하는 표정을 짓습니다.

### C. 전반적인 톤 앤 매너 (Tone & Manner)
- **컬러 스킴**: 딱딱한 다크 모드(IDE 느낌)보다는, 파스텔 톤의 부드러운 배경이나 크림색/오트밀 계열을 사용하여 갤러리와 캔버스의 따뜻함을 줍니다. UI의 그림자도 둥글고 팝한 느낌(Neumorphism 또는 Soft Pop)을 적용합니다.
- **마이크로 애니메이션**: 버튼을 누를 때 젤리처럼 눌리거나, 성공 시 꽃가루 파티클이 터지는 등 만지기 즐거운(Playful) UI를 지향합니다.

## 9. 서비스 네이밍 및 타이틀
- **내부 프로젝트명**: **Code2Art** (리포지토리 및 코드베이스 네이밍)
- **외부 서비스 타이틀**: **Is My Code Art?** (약칭: IMCA)
- **도메인 및 마케팅 컨셉**: 방문자(개발자)에게 도발적이면서도 철학적인 질문을 던지는 문장형 도메인입니다. SNS에 `[내 코드는 예술일까? - IsMyCode.Art]` 와 같은 링크를 공유하는 것 자체만으로도 동료 개발자들의 즉각적인 호기심과 도전을 유발하여 자발적인 바이럴 루프를 만들어냅니다.
